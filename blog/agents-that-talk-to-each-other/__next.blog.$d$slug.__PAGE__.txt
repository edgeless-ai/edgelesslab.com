4:T1c54,<p>I run five AI agents concurrently. A dispatch agent on my Mac routes tasks. Worker agents on a VPS handle execution. They communicate through two channels: a real-time message bus for urgent coordination, and async inboxes for everything else.</p>
<p>This isn't a framework demo. These agents process real content, manage real infrastructure, and occasionally make real mistakes. The architecture exists because simpler approaches failed first.</p>
<h2 id="why-single-agent-breaks-down">Why Single-Agent Breaks Down</h2>
<p>A single Claude Code session can do remarkable things. But it has limits: one context window, one set of tools, one conversation thread. When you need to research while building while monitoring, a single agent becomes a bottleneck.</p>
<p>The first instinct is to make the agent smarter. Give it more tools. Expand its context. Write better prompts. This works until it doesn't. The failure mode is subtle: the agent starts losing track of parallel concerns. It forgets it was monitoring something while deep in a code change. Context compression drops the monitoring task.</p>
<p>The fix isn't a smarter agent. It's more agents with clear boundaries.</p>
<h2 id="the-dispatch-worker-topology">The Dispatch/Worker Topology</h2>
<div class="blog-viz blog-flow"><div class="blog-viz-title">Dispatch/Worker Architecture</div><div class="blog-flow-row"><span class="blog-flow-box">Human</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Dispatch</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Code Worker</span></div><div class="blog-flow-row"><span class="blog-flow-box">Dispatch</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Research Worker</span></div><div class="blog-flow-row"><span class="blog-flow-box">Dispatch</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Monitor Worker</span></div><div class="blog-flow-row"><span class="blog-flow-box">Dispatch</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Content Worker</span></div></div>
<p>One agent dispatches. The rest execute. The dispatch agent has a complete view of what needs to happen. Worker agents have narrow focus and specific tool sets.</p>
<p>The dispatch agent doesn't write code. It doesn't browse the web. It doesn't manage infrastructure. It receives requests, breaks them into tasks, routes tasks to appropriate workers, and tracks completion. Its only tools are: send message, check status, read results.</p>
<p>Workers are specialists. One handles code changes. One handles research. One monitors infrastructure. One processes content. They don't talk to each other directly. All coordination flows through dispatch.</p>
<p>This is a deliberate constraint. Peer-to-peer agent communication creates the same problems as distributed systems: race conditions, split-brain states, cascading failures. A central dispatcher eliminates these by serializing decisions.</p>
<h2 id="two-communication-channels">Two Communication Channels</h2>
<p><strong>Agent Bus (real-time)</strong>: A message hub running as a launchd service on port 9800. Agents connect via MCP, send typed messages, and receive responses. Messages are held for 24 hours if the recipient is offline. The bus handles: task assignments, status updates, urgent alerts, and result delivery.</p>
<p>The bus is fire-and-forget from the sender's perspective. You send a message, the hub routes it. If the recipient is connected, delivery is instant. If not, it queues. Senders don't block waiting for responses.</p>
<p><strong>Async inboxes (batch)</strong>: A file-based system synced between machines via rsync every 60 seconds. Each agent has an inbox directory. Dispatch writes task files. Workers read them, process, and write results to an outbox. A daemon picks up results and routes them back.</p>
<p>Why two channels? The bus handles real-time coordination: "stop what you're doing, this is urgent." Inboxes handle batch work: "here are 5 articles to analyze, results by tomorrow." Using one channel for both creates priority inversion. The urgent message sits behind 50 batch tasks.</p>
<h2 id="state-machine-per-task">State Machine Per Task</h2>
<div class="blog-viz blog-flow"><div class="blog-viz-title">Task State Machine</div><div class="blog-flow-row"><span class="blog-flow-box">Queued</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Acked</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Running</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Done</span></div><div class="blog-flow-row"><span class="blog-flow-box">Running</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Failed</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Retry</span><span class="blog-flow-arrow">→</span><span class="blog-flow-box">Queued</span></div></div>
<p>Every task follows a state machine: <code>queued -> acked -> running -> done | failed</code>. Dispatch creates tasks in <code>queued</code>. Workers acknowledge with <code>acked</code> (proving they received it). Work begins at <code>running</code>. Terminal states are <code>done</code> (with results) or <code>failed</code> (with error context).</p>
<p>The state machine solves the "did it even start?" problem. If a task stays in <code>queued</code> for more than 5 minutes, dispatch knows the worker is down. If it stays in <code>running</code> for too long, dispatch can reassign or escalate. Without explicit states, you're guessing.</p>
<p>Failed tasks include structured error context: what went wrong, whether it's retryable, and what the worker tried. Dispatch uses this to decide: retry with the same worker, route to a different worker, or escalate to a human.</p>
<h2 id="what-most-frameworks-get-wrong">What Most Frameworks Get Wrong</h2>
<p>Multi-agent frameworks love complex abstractions. Shared memory. Consensus protocols. Agent hierarchies with managers and sub-managers. These solve theoretical problems at the cost of operational simplicity.</p>
<p>The problems that actually matter in production are boring: How do you know if an agent crashed? How do you restart a failed task? How do you add a new worker type without changing the dispatch logic? How do you debug a task that produced wrong results?</p>
<p>A simple dispatch/worker topology with explicit state machines and two communication channels answers all of these. It's not elegant. It doesn't make a good conference talk. But it runs unattended at 3am without surprises.</p>
<h2 id="getting-started">Getting Started</h2>
<p>You don't need five agents on day one. Start with two: one dispatch, one worker. Route one type of task. Get the communication channel working. Add the state machine. Only then add a second worker type.</p>
