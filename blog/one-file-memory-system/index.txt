23:T14d4,<p>Every session, I used to start the same way. "We're using TypeScript, not JavaScript." "Don't use default exports." "The API is in <code>src/api/</code>, not root." "We already tried Redis here and it didn't work."</p>
<p>Five minutes of throat-clearing before any real work happened. Every. Single. Session.</p>
<p>Then I set up a memory file, and that problem disappeared.</p>
<h2 id="the-pain-point">The Pain Point</h2>
<p>Claude Code is stateless by design. Every session starts fresh. There's no session history, no learned preferences, no memory of last week's architecture decision. This isn't a bug; it's a consequence of how the model works. But it creates real friction.</p>
<p>The compounding effect is the worst part. Every correction you make in one session is a correction you'll make again next week. You're not building on previous sessions; you're re-establishing context every time.</p>
<p>This is especially painful with project-specific knowledge: "Don't touch the authentication middleware, it's under active refactor." "The staging database is read-only." "We deploy from the <code>release</code> branch, not <code>main</code>."</p>
<h2 id="the-simplest-possible-fix">The Simplest Possible Fix</h2>
<p>Claude Code reads a file called <code>CLAUDE.md</code> at session start. That's the hook. Put things in that file that Claude should always know, and it will always know them.</p>
<p>A minimal <code>CLAUDE.md</code> solves 80% of the problem:</p>
<pre><code># Project: My App

## Stack
- TypeScript (strict mode), React 19, Next.js 16
- Postgres via Drizzle ORM
- Vitest for tests (no Jest)
- Tailwind CSS

## Conventions
- No default exports
- API lives in src/api/
- Tests colocated with source files

## Don't Do
- Don't use mocks in integration tests; hit the real DB
- Don't add inline styles; use Tailwind classes</code></pre>
<p>That's it. Three sections. Less than 20 lines. Claude reads it at session start and you never repeat those instructions again.</p>
<h2 id="the-before-after">The Before/After</h2>
<p><strong>Before memory:</strong></p>
<blockquote>Me: "Let's add a new API endpoint."<br/>Claude: *writes a JavaScript file with a default export and Jest tests*<br/>Me: "TypeScript, no default exports, Vitest please."<br/>Claude: *rewrites*<br/>(Repeat every session)</blockquote>
<p><strong>After memory:</strong></p>
<blockquote>Me: "Let's add a new API endpoint."<br/>Claude: *writes TypeScript, named exports, Vitest tests, in <code>src/api/</code>*<br/>(Never needs to be said again)</blockquote>
<p>After a month of accumulated memory, I tracked roughly 60% fewer correction cycles per session. Not a formal benchmark, just counting how often I typed "I already told you that."</p>
<h2 id="the-memory-file-pattern">The Memory File Pattern</h2>
<p>A single <code>CLAUDE.md</code> works. But once you start accumulating more context, a simple structure helps.</p>
<p>The pattern I use across projects on this system (documented in detail in the <a href="/products">Claude Memory Kit</a>):</p>
<p><strong>User memory</strong>: who you are and how you work. Goes in your home directory CLAUDE.md so it follows you across every project. Things like: "I'm a backend engineer who's new to React. Explain frontend patterns using backend analogies."</p>
<p><strong>Feedback memory</strong>: corrections that stick. When Claude does something wrong and you correct it, add that correction to a memory file. It becomes permanent. "Don't use try-catch in React components; use error boundaries."</p>
<p><strong>Project memory</strong>: architecture decisions, frozen APIs, deployment conventions. Project-specific.</p>
<p><strong>Reference memory</strong>: where things live. "Staging environment: staging.myapp.com. Admin dashboard: Linear workspace 'Platform'."</p>
<h2 id="set-it-up-in-10-minutes">Set It Up in 10 Minutes</h2>
<ol><li>Create <code>CLAUDE.md</code> in your project root</li><li>Add your stack, 3-5 conventions, and 2-3 "never do this" rules</li><li>Start a new Claude Code session (it will read the file automatically)</li><li>For the first few sessions, notice when Claude gets something wrong. Add that correction to the file</li><li>After a week, the file has become a trained reflex</li></ol>
<p>The free version of the <a href="https://github.com/edgeless-ai/claude-memory-kit">Claude Memory Kit</a> includes templates for all four memory types and a starter CLAUDE.md structure. If you want stack-specific libraries and multi-project memory patterns, the <a href="/products">Pro version</a> covers those.</p>
<h2 id="one-more-thing">One More Thing</h2>
<p>Memory files do accumulate cruft. Review monthly. Archive anything that's no longer true. Keep each file under 200 lines. Memory that's too long wastes context window on stale instructions.</p>
<p>The discipline: when you update your architecture, update your memory file the same day. It takes 30 seconds, and it means next session Claude already knows.</p>
<p>That 10-minute setup has probably saved me 10 hours over the past few months. It's the highest-leverage thing I've done to improve how I work with Claude Code.</p>
