4:T15a0,<p>Demo prompts work great in demos. "Summarize this article" returns a clean summary. "Extract the key entities" returns a nice list. Ship that to production and watch it break on the first malformed input.</p>
<p>The gap between demo prompts and production prompts is the same gap between a script and a system. One handles the happy path. The other handles everything.</p>
<h2 id="the-three-failure-modes">The Three Failure Modes</h2>
<p>Production prompts fail in predictable ways. Once you know the patterns, you can design against them.</p>
<p><strong>Drift</strong>: the model's interpretation of your prompt shifts as context accumulates. A prompt that works perfectly in message 1 starts hallucinating by message 15 because earlier responses have polluted the context. Fix: restate critical constraints at decision points, not just at the top.</p>
<p><strong>Edge collapse</strong>: the model encounters an input it wasn't designed for and produces confidently wrong output instead of signaling uncertainty. The classic: a sentiment classifier that labels gibberish as "positive" because it always picks something. Fix: give the model an explicit "I can't classify this" option and define when to use it.</p>
<p><strong>Format rot</strong>: the model returns valid content in the wrong structure. You asked for JSON, it returns JSON with markdown wrapping. You asked for a list, it returns a paragraph with embedded list items. Fix: provide a concrete output example, not just a format description.</p>
<h2 id="structural-patterns-that-work">Structural Patterns That Work</h2>
<p>After writing hundreds of production prompts across classification, extraction, generation, and analysis tasks, a few structural patterns consistently outperform.</p>
<p><strong>The constraint sandwich</strong>: state the task, list constraints, restate the most critical constraint. Models weight the end of the prompt more heavily. If "never include PII" is your most important constraint, say it last.</p>
<p><strong>Explicit refusal criteria</strong>: tell the model exactly when to say "I don't know" or "this input doesn't match." Without this, models will always produce something, even when the right answer is nothing.</p>
<p><strong>Output scaffolding</strong>: provide the exact structure you expect, with placeholders. Not "return JSON with the fields name, score, and reasoning" but:</p>
<pre><code>{
  "name": "...",
  "score": 0-10,
  "reasoning": "One sentence explaining the score"
}</code></pre>
<p>The model mirrors structure more reliably than it follows structural descriptions.</p>
<p><strong>Temperature as a design parameter</strong>: temperature 0 for extraction and classification. Temperature 0.3-0.7 for generation where variety matters. Temperature 1.0+ only for brainstorming where you want surprise. Most production tasks should be at 0.</p>
<h2 id="testing-prompts-like-code">Testing Prompts Like Code</h2>
<p>The mistake most teams make: they test prompts manually, with their own inputs, against their own expectations. This is like testing a function by calling it once with the example from the README.</p>
<p>Production prompt testing needs:</p>
<p><strong>Edge cases as fixtures</strong>: empty input, extremely long input, input in the wrong language, input with injection attempts, input that contradicts the prompt's assumptions. Build a test suite of these and run every prompt revision against all of them.</p>
<p><strong>Regression tracking</strong>: when you improve a prompt for one case, you need to know if other cases degraded. An A/B comparison template that runs both versions against the full test suite and diffs the outputs.</p>
<p><strong>Scoring rubrics</strong>: not "did it work?" but "did it score 8+ on accuracy, 7+ on format compliance, and 6+ on reasoning quality?" Structured scoring catches subtle degradation that pass/fail misses.</p>
<p>The <a href="/products">Prompt Testing Framework</a> includes templates for all three of these patterns, pre-built for Claude, GPT, and Gemini.</p>
<h2 id="the-maintenance-angle">The Maintenance Angle</h2>
<p>Prompts are code. They need versioning, review, and documentation.</p>
<p>Every prompt in our system includes:</p>
<ul><li>A version number</li><li>A one-line purpose statement</li><li>The last date it was tested against the full edge case suite</li><li>The model and temperature it was designed for</li></ul>
<p>When a model updates (GPT-4 to GPT-4o, Claude 3 to Claude 4), every prompt gets retested. Model updates change prompt behavior in subtle ways; a prompt that worked perfectly on Claude 3.5 might need adjustment on Claude 4 because the model's default behavior shifted.</p>
<h2 id="the-checklist">The Checklist</h2>
<p>Before shipping a prompt to production:</p>
<ol><li>Have you tested with empty/null input?</li><li>Have you tested with adversarial input?</li><li>Does the model have an explicit "I can't do this" path?</li><li>Is the output format specified by example, not description?</li><li>Are critical constraints restated at the end of the prompt?</li><li>Is the temperature appropriate for the task type?</li><li>Have you run a regression against the previous prompt version?</li></ol>
<p>If any answer is "no," the prompt isn't production-ready. It's a demo.</p>
